Sobrecarga > 95%

Demo 2
Simula una sobrecarga crítica en un distrito. El backend detecta el consumo ≥95%, crea una alerta SOBRECARGA_CRITICA y sugiere redistribución de carga hacia distritos con menos del 75% de uso.
⚙️

Hora Pico y Escalado

Demo 3
Activa el modo hora pico en el simulador (burst de datos 3x). Luego escala el backend a 2 instancias con Docker. El frontend no se desconecta gracias a Redis pub/sub y Nginx.
docker compose up --scale energygrid-backend=2 -d
docker compose up -d
1. Activa hora pico → 2. Escala en terminal → 3. Verifica 2 instancias con docker ps → 4. El frontend sigue funcionando → 5. Desactiva hora pico → 6. Vuelve a 1 backend
🏗️

Caída de Subestación

Demo 4
Simula la desconexión de una subestación. El watchdog del backend detecta que no recibe datos por 15 segundos, crea una alerta SUBESTACION_DESCONECTADA y sugiere redistribución.
⏱️ Espera ~15 segundos después de detener para que el watchdog detecte la caída.
🕐

Timestamps Inválidos

Demo 5
Envía datos con timestamp fuera del rango permitido (>24h en el futuro o pasado). El backend rechaza la petición con HTTP 422 y registra el evento TIMESTAMP_INVALIDO en los logs.
docker compose logs energygrid-backend | grep TIMESTAMP_INVALIDO
🛡️

SQL Injection

Demo 6
Envía un payload malicioso con '; DROP TABLE consumo_temporal; -- El backend detecta el patrón sospechoso con regex, marca la anomalía, usa queries parametrizadas que lo tratan como string literal inofensivo, y registra sospecha_sql en los logs.
docker compose logs energygrid-backend | grep sospecha_sql
📊

Dashboard de Monitoreo

Demo 7
Muestra las 3 herramientas de monitoreo integradas: Munin (métricas del sistema), Pingdom (disponibilidad de endpoints) y Slow Query Log (pg_stat_statements).
Endpoints de la API de monitoreo:
curl http://localhost:8000/api/monitoring/health
curl http://localhost:8000/api/monitoring/dashboard
curl http://localhost:8000/api/monitoring/alerts
curl http://localhost:8000/api/monitoring/pingdom/status
curl http://localhost:8000/api/monitoring/queries/slow
💾

Persistencia y Tolerancia a Fallos

Demo 8
Demuestra que el sistema mantiene los datos ante fallos. Si la BD cae, el backend encola los datos en memoria y los reinserta al recuperarse. Los datos persisten incluso después de reiniciar contenedores.
Caída de BD con simulador activo:
docker compose stop energygrid-db
docker compose logs energygrid-backend | grep queue
docker compose start energygrid-db
Persistencia de datos históricos:
docker compose down
MSYS2_ARG_CONV_EXCL="*" docker compose up -d
curl http://localhost:8000/api/districts
⚠️ Los datos históricos persisten porque PostgreSQL usa volúmenes Docker.
